# /learn — 選択・感想の学習

ユーザーがプランページの「選択をコピー」で貼った内容（`/learn ...` の行＋JSON）を解析し、学習資産を更新する。
JSON 形式: `{ "weekend": "2026-07-04", "likes": ["a","b"], "went": "b" | "none" | null, "memo": "..." }`
- `likes` = いいねした案（複数可・0個も可）。**提案自体の良さ**のシグナル。
- `went` = 実際に行った案1つ。`"none"`=行かなかった、`null`=未回答。

## 手順

1. **plans.json の feedback を記入**: 該当 `weekend` の `feedback` に `likes` / `went` / `memo`（rating があれば rating も）を書き込む。

2. **taste-profile.md の選択ログに1行追記**: 週末・提示3案の要約・いいね・行った案・感想の要点・読み取れた好み。
   - **likes は「その案の方向性が刺さった」証拠**として仮説に反映（行かなくても価値がある）。
   - 感想から好みの仮説が読み取れたら「仮説」節に追加または既存仮説に証拠を追記。
   - **仮説が3回以上支持されたら「確定した好み」へ昇格**させる（昇格日と証拠3件を書く）。
   - 逆に不支持の証拠が出たら仮説に反証として記録する。

3. **visited.json に登録**（`went` が "none"/null 以外の場合のみ）: 行った案の主目的地・エリアを追記。
   形式例: `{ "date": "2026-07-05", "spot": "高尾山", "area": "高尾", "kind": "hike", "from": "2026-07-05-a" }`
   これが次回以降の「過去4週の重複回避」と風の枠の「未訪問」判定に使われる。

4. **とっておきの確認**: `likes` に含まれ `went` でない案は「とっておき」（後日 /plan が時季・天気の合う週に再提案してよい）。特別な台帳は不要——plans.json の feedback がそのまま台帳になる。

5. **wishlist.md の消し込み**: 行った内容が wishlist の未実施（☐）に該当するなら ☑ に更新。新しい「やりたい」が memo にあれば追記。

6. **commit → push** して配信ページのアーカイブ（現像済みアルバム）に反映させる。「現像済」スタンプは `went` が案レターのとき点灯し、いいねシールもアーカイブに残る。

## 注意

- 感想の解釈で断定しすぎない。1回の発言は仮説どまり（3回ルール）。
- 夫婦で好みが割れている兆候（例: お酒系）は個人差としてメモし、共有プロファイルには控えめに反映。
